Background

DC motors with closed-loop control systems provide some of the fastest, most efficient, and most accurate systems available for moving things to specified positions. A motor with closed-loop control is often referred to as a servo system. If you have a motor, motor driver and position sensor just lying around, you can write code to operate the motor in closed-loop position servo mode. One algorithm is the proportional controller:

  • Supply as an input the setpoint, the desired location of the motor.

  • Subtract the measured location of the motor from the setpoint; the difference is the error signal, a signed number indicating which way the motor is off and how far.

  • Multiply the error signal by a control gain called KP to produce a result called the actuation signal. The larger the error, the larger the actuation, so the harder the controller will push. The formula is

  • Send the actuation signal to the motor driver which you have already written to control the magnitude and direction of motor torque.

  • You may add other terms based on the time integral and time derivative of the error, or even full state feedback using the position and velocity to compute the actuation, but such fancy stuff is not required for this exercise.

The units of gains such as KP can be weird. If position is measured by an encoder in units of encoder counts and the actuation signal is a duty cycle in percent, the units of KP are percent duty cycle per encoder count. Really.

An often challenging task is tuning the controller â€“ adjusting operating parameters such as KP so that the controller pushes the motor hard enough to get the load to the correct place quickly and accurately, but not so hard as to cause excessive overshoot or instability. To assist with controller tuning, we often perform step response tests in which the setpoint is instantly changed from zero to some nonzero value and the response of the system is recorded. The response is usually plotted on a graph of motor position vs. time.

Trick: Dual USB-Serial Ports

It can be helpful to use two USB cables for a lab such as this:

  • One to run and debug your program as usual

  • Another to transmit data from your MicroPython program to your CPython program running on a PC

A procedure for this is the following:

  • Create a file boot.py on your MicroPython board which contains the following code:

    import pyb                  # Turn off the REPL on UART2
pyb.repl_uart(None)

  • Soft-restart the Nucleo with Ctrl-D and ensure everything works normally

  • Find the name of the existing USB-serial port:

    • On Windowsâ„¢, check your device manager for a serial port named COMx to find the number x

    • On MacOSâ„¢ or Linux, check the /dev/ folder for a file called /dev/tty.usbserialxyz or /dev/ttyACMx and note the number

  • Plug in a second mini-USB cable to the Nucleo (above the cable which is already connected to the Shoe) and your PC. You may need a hub or extra dongle

  • Use the device manager or /dev/ folder to find the name/number of the newly added serial port

  • Keeping your Python development system (Thonny or other) running, start a terminal program such as PuTTY, and have it connect to the newly added serial port

  • In your software, write data such as times and motor positions to the second USB-serial port, which is UART2 to MicroPython:

    u2 = pyb.UART(2, baudrate=115200)      # Set up the second USB-serial port

for number in range(10):               # Just some example output
    u2.write(f"Count: {number}\r\n")   # The "\r\n" is end-of-line stuff
    number += 1

  • Thonny may have trouble automatically detecting the Shoe serial port; if so, in Runâ†’Select Interpreter... tell it the name of the first serial port you found before connecting the second one
